# Техническое задание (ТЗ)

## 1. Общая информация о проекте

**Название проекта:**  
DocuMind — AI PDF Contract Analyzer

**Формат проекта:**  
Локальный сервис для обработки документов с интеграцией с внешними API

**Тип продукта:**  
Backend-приложение с двумя пользовательскими интерфейсами:
- Telegram-бот
- Web UI

**Цель проекта:**  
Разработка MVP-сервиса для автоматизации обработки PDF-договоров, включая извлечение данных, анализ содержания и генерацию структурированного результата

**Срок реализации:**  
1–2 недели

**Технологический стек (предварительно):**  
- Python  
- FastAPI  
- Telegram Bot API (интерфейс взаимодействия с пользователем)  
- LLM API (через proxyapi)  
- работа с PDF (pdfplumber / PyMuPDF)
- OCR (pytesseract / easyocr или аналогичный инструмент)    
- генерация DOCX (python-docx)  
- интеграция с Google Drive API  

**Дополнительные требования:**  
- реализация логирования процесса обработки  
- отображение статусов выполнения задачи пользователю через Telegram-бота  
- вывод технических логов в консоль для отладки  

**Формат разработки:**  
Индивидуальный учебный проект

---

## 2. Цели и задачи проекта

### 2.1. Основная цель

Разработать MVP-сервис, который позволяет автоматически обрабатывать PDF-договоры, извлекать из них структурированные данные, формировать краткий аналитический отчет и генерировать итоговый документ, минимизируя ручной труд пользователя.

---

### 2.2. Задачи проекта

Для достижения цели необходимо реализовать следующие задачи:

- реализовать загрузку PDF-документа через интерфейс Telegram-бота  
- реализовать OCR-обработку для сканированных PDF-документов  
- извлекать текст из PDF-документа  
- реализовать локальную деперсонализацию чувствительных данных  
- интегрировать LLM через proxyapi для анализа текста  
- реализовать извлечение ключевых параметров договора  
- сформировать краткий аналитический отчет по документу  
- реализовать генерацию итогового файла в формате `.docx`  
- реализовать систему статусов обработки в Telegram  
- реализовать логирование процесса обработки в консоль  
- реализовать опциональное сохранение результатов в Google Drive
- реализовать замер времени выполнения основных этапов обработки  

---

### 2.3. Ключевые метрики (KPI)

- успешная обработка документа без критических ошибок
- корректное извлечение ключевых данных (визуальная проверка)  
- корректное формирование итогового `.docx` файла  
- наличие понятных статусов обработки для пользователя  
- отсутствие хранения персональных данных после завершения обработки  
- успешная загрузка результата в Google Drive (при включенной опции)  
- измерение и логирование времени выполнения обработки документа
- соответствие времени обработки заданному диапазону (до 60–120 секунд)
- успешная обработка сценария сравнения двух документов  

---

### 2.4. Критерии успешности MVP

MVP считается успешным, если:

- пользователь может загрузить PDF-договор через Telegram-бота  
- система корректно обрабатывает документ и не падает  
- пользователь получает:
  - аналитический отчет  
  - итоговый `.docx` файл  
- процесс сопровождается понятными статусами  
- данные не сохраняются после завершения обработки  

---

## 3. Функциональные требования

### 3.1. Общий сценарий работы системы

Пользователь взаимодействует с системой через:

- Telegram-бота  
- Web-интерфейс

Основной сценарий работы:

1. Пользователь выбирает режим работы:
   - распознавание договора  
   - анализ договора  
   - сравнение двух договоров  
2. В зависимости от режима пользователь загружает один или два документа (PDF или DOCX)  
3. Система принимает файл и уведомляет пользователя о начале обработки  
4. Выполняется извлечение текста из PDF:
   - если документ содержит текстовый слой — используется стандартное извлечение  
   - если текстовый слой отсутствует — применяется OCR    
5. Выполняется локальная деперсонализация данных  
6. Обезличенный текст передается в LLM через proxyapi  
7. Выполняется анализ документа и извлечение ключевых данных  
8. Формируется краткий аналитический отчет  
9. Генерируется итоговый файл в формате `.docx`  
10. (Опционально) результат сохраняется в Google Drive  
11. Пользователь получает результат и уведомление о завершении обработки  

---

### 3.2. Прием и обработка PDF-документа

Система должна:

- принимать PDF-файл от пользователя через Telegram  
- проверять формат файла (поддержка только PDF)  
- обеспечивать обработку документа преимущественно в оперативной памяти с возможным использованием временного хранения на диске в рамках MVP допускается сохранение промежуточных данных в директории `/data`, включая:
  - временные результаты обработки
  - служебные данные (lock, OAuth токены)
- извлекать текст из PDF-документа  
- определять наличие текстового слоя в документе  
- при отсутствии текстового слоя применять OCR для извлечения текста    

---

### 3.3. Деперсонализация данных

Система должна:

- выполнять локальную обработку текста документа  
- выявлять персональные и чувствительные данные (ФИО, реквизиты, адреса, контакты и т.д.)  
- заменять их на ролевые или технические маркеры (например: "Заказчик", "Исполнитель", "ИНН_1")  
- не передавать исходные персональные данные в LLM  

---

### 3.4. Анализ документа с помощью LLM

Система должна:

- отправлять обезличенный текст документа в LLM через proxyapi  
- извлекать ключевые параметры договора, включая:
  - стороны договора (в обезличенном виде)  
  - предмет договора  
  - сроки  
  - суммы  
  - основные обязательства  
- формировать структурированные данные (например, JSON)  

---

### 3.5. Генерация аналитического отчета

Система должна:

- формировать краткий аналитический отчет по содержанию договора  
- включать в отчет:
  - краткое описание документа  
  - ключевые условия  
  - потенциально важные или спорные моменты  
- предоставлять отчет пользователю в текстовом виде
- система должна отображать пользователю дисклеймер о том, что отчет носит справочный характер и не является юридической консультацией    

---

### 3.6. Генерация итогового документа

Система должна:

- формировать файл в формате `.docx`  
- использовать структурированные данные для заполнения документа  
- обеспечивать читаемую и понятную структуру итогового файла  

---

### 3.7. Интеграция с Google Drive

Система должна:

- предоставлять пользователю возможность выбора сохранения результата в Google Drive перед началом обработки
- обеспечивать возможность сохранения результата в Google Drive  
- использовать ранее реализованный модуль интеграции  
- создавать файл в указанной папке пользователя  
- возвращать пользователю ссылку на сохраненный файл  
- реализовано разделение namespace для разных интерфейсов:
  - `/google-drive/...` — для Web UI
  - `/google-drive-bot/...` — для Telegram-бота

---

### 3.8. Отображение статусов обработки

Система должна:

- информировать пользователя о ходе обработки документа  
- отображать основные этапы обработки в зависимости от выбранного режима  
- обеспечивать понятную последовательность сообщений в Telegram  

Для режима распознавания договора система должна отображать статусы:

- получение файла  
- извлечение текста (или запуск OCR при необходимости)  
- деперсонализация данных  
- анализ документа  
- формирование отчета  
- генерация `.docx`  
- сохранение в Google Drive (если выбрана опция)  
- завершение обработки  

Для режима анализа договора система должна отображать статусы:

- получение файла  
- извлечение текста (или запуск OCR при необходимости для PDF)  
- деперсонализация данных  
- анализ документа  
- формирование аналитического отчета  
- сохранение результата в Google Drive (если выбрана опция)  
- завершение обработки  

Для режима сравнения двух договоров система должна отображать статусы:

- получение документов  
- извлечение текста из первого документа (или OCR при необходимости)  
- извлечение текста из второго документа (или OCR при необходимости)  
- деперсонализация данных  
- сравнительный анализ документов  
- формирование сравнительного отчета  
- подготовка распознанных результатов (если выбрана опция возврата)  
- отправка результатов в чат или сохранение в Google Drive  
- завершение обработки  

---

### 3.9. Логирование и мониторинг

Система должна:

- выводить технические логи в консоль  
- фиксировать выбранный режим работы системы  
- фиксировать формат входных документов  
- фиксировать выбранный способ извлечения текста (прямое извлечение или OCR)  
- фиксировать факт использования OCR для каждого документа  
- фиксировать этапы выполнения обработки  
- измерять время выполнения каждого этапа  
- фиксировать общее время обработки документа или пары документов  
- логировать ошибки обработки и внешних интеграций  
- фиксируются метрики выполнения сценариев (processing_metrics), включая:
  - время выполнения этапов обработки
  - общее время обработки
- метрики используются для оценки производительности MVP

---

### 3.10. Ограничения системы

Система должна:

- обрабатывать только PDF-документы  
- не сохранять персональные данные  
- не хранить историю обработки  
- работать в stateless-режиме
- используется механизм блокировки обработки (lock), допускающий выполнение только одного сценария обработки одновременно
- при наличии активной обработки новые запросы отклоняются с уведомлением пользователя  

---

### 3.11 Режимы работы системы

Система должна поддерживать следующие режимы:

1. Распознавание и структурирование договора  
   - вход: PDF  
   - выполняется извлечение текста (или OCR)  
   - формируется структурированный результат и `.docx`  

2. Анализ договора  
   - вход: PDF или DOCX  
   - выполняется анализ содержания  
   - формируется аналитический отчет  

3. Сравнение двух договоров  
   - вход: два документа (PDF или DOCX)  
   - выполняется извлечение текста (или OCR)  
   - выполняется сравнительный анализ  
   - формируется отчет с различиями  
  
---

## 4. Нефункциональные требования

### 4.1. Производительность

- время обработки одного документа должно составлять не более 60–120 секунд  
- система должна корректно обрабатывать документы среднего объема (до ~20–30 страниц)  
- система должна обеспечивать последовательную обработку запросов без критических задержек
- - система протестирована в условиях ограниченных ресурсов (~1GB RAM)
- при высокой нагрузке используется swap
- производительность зависит от:
  - размера документа
  - необходимости OCR
  - времени ответа LLM  

---

### 4.2. Надежность и стабильность

- система не должна завершаться с ошибкой при корректных входных данных  
- при возникновении ошибки система должна:
  - логировать ошибку в консоль  
  - уведомлять пользователя о проблеме  
- система должна корректно обрабатывать невалидные входные данные (например, не-PDF файлы)  

---

### 4.3. Безопасность

- система не должна сохранять персональные данные пользователей  
- обработка данных должна выполняться в оперативной памяти  
- перед передачей данных в LLM должна выполняться деперсонализация  
- исходные данные не должны передаваться во внешние сервисы без обработки  

---

### 4.4. Масштабируемость (ограниченная для MVP)

- система рассчитана на обработку одного запроса за раз. Реализовано через механизм shared lock на базе SQLite 
- многопользовательская и параллельная обработка не является целью MVP
  

---

### 4.5. Удобство использования (UX)

- пользователь должен получать понятные статусы обработки в Telegram  
- сообщения должны быть последовательными и информативными  
- пользователь должен понимать текущий этап обработки  

---

### 4.6. Поддерживаемость и расширяемость

- проект должен быть реализован в модульной архитектуре  
- каждая функциональная часть должна быть вынесена в отдельный модуль  
- код должен быть структурирован и читаем  
- должна быть возможность расширения функционала (например, добавление новых типов документов)  

---

### 4.7. Ограничения

- система работает только с PDF-документами  
- система не выполняет юридическую проверку документов  
- аналитический отчет носит справочный характер и не является юридическим заключением
- система не гарантирует 100% точность извлечения данных  

---

## 5. Архитектура системы

### 5.1. Общий подход

Система проектируется как модульное приложение, в котором каждая ключевая подзадача выделена в отдельный модуль. Такой подход должен обеспечить:

- читаемость и поддерживаемость кода  
- упрощение тестирования  
- возможность поэтапной доработки  
- возможность повторного использования отдельных компонентов в других проектах  

Система включает два основных уровня:

- **интерфейсный уровень** — взаимодействие пользователя с системой через Telegram-бота  
- **сервисный уровень** — обработка документа, анализ, генерация результата и интеграции  

---

### 5.2. Основные модули системы

Планируется выделить следующие модули:

#### 1. Telegram Bot Module
Отвечает за:
- прием файлов от пользователя  
- отправку пользователю статусов обработки  
- отправку итоговых результатов  
- вывод дисклеймера по аналитическому отчету  

#### 2. File Handling Module
Отвечает за:
- прием PDF-файла  
- временную работу с файлом в рамках обработки  
- передачу содержимого в следующие этапы пайплайна  

#### 3. PDF Text Extraction Module
Отвечает за:
- извлечение текста из PDF-документа  
- подготовку текста к дальнейшей обработке  

#### 3.1 OCR Module
Отвечает за:
- обработку сканированных PDF-документов  
- извлечение текста из изображений страниц  
- передачу результата в модуль обработки текста  
  
#### 4. Data Masking Module
Отвечает за:
- локальную деперсонализацию документа  
- замену персональных и чувствительных данных на маркеры и ролевые обозначения  
- исключение передачи исходных данных в LLM  

#### 5. LLM Analysis Module
Отвечает за:
- взаимодействие с LLM через proxyapi  
- извлечение ключевых параметров договора  
- формирование структурированного результата  
- генерацию краткого аналитического отчета  

#### 6. Document Generation Module
Отвечает за:
- формирование итогового `.docx`-файла  
- сбор итогового результата для пользователя  

#### 7. Google Drive Integration Module
Отвечает за:
- сохранение итоговых файлов в Google Drive  
- использование ранее реализованных модулей интеграции  

#### 8. Logging and Metrics Module
Отвечает за:
- вывод логов в консоль  
- фиксацию этапов обработки  
- замер времени выполнения каждого этапа  
- фиксацию общего времени обработки документа  

#### 9. Orchestration Module
Отвечает за:
- последовательный запуск этапов обработки  
- управление общим пайплайном  
- обработку ошибок  
- возврат результата пользователю  

---

### 5.3. Общий сценарий взаимодействия модулей

Поток обработки документа должен выглядеть следующим образом:

1. Telegram Bot Module принимает PDF-файл от пользователя  
2. File Handling Module подготавливает файл к обработке  
3. PDF Text Extraction Module:
   - пытается извлечь текст из PDF  
   - при отсутствии текста инициирует OCR Module  
4. Data Masking Module выполняет деперсонализацию  
5. LLM Analysis Module анализирует обезличенный текст  
6. Document Generation Module формирует итоговый `.docx`  
7. Google Drive Integration Module (при необходимости) сохраняет результат  
8. Telegram Bot Module отправляет пользователю:
   - аналитический отчет  
   - итоговый файл  
   - статус завершения обработки  

---

### 5.4. Архитектурные принципы

При разработке системы должны соблюдаться следующие принципы:

- модульность  
- разделение ответственности  
- отсутствие хранения персональных данных  
- stateless-подход к обработке  
- минимизация связности между модулями  
- возможность замены отдельных компонентов без полной переработки системы  

---

### 5.5. Ограничения архитектуры MVP

Для MVP:

- не реализуется многопользовательская конкурентная обработка  
- не реализуется база данных для хранения истории документов  
- не реализуется полноценная система ролей и авторизации 
- отсутствует очередь задач (используется lock-механизм)
- отсутствует горизонтальное масштабирование
- обработка выполняется синхронно 

---

### 5.6. Фактическая архитектура MVP

В рамках реализации проект развернут на двух серверах:

- **Server 1 (backend)**:
  - FastAPI (dm-api)
  - Telegram-бот (dm-bot)

- **Server 2 (frontend)**:
  - Web UI (dm-web)
  - Nginx (reverse proxy + SSL)

Все интерфейсы используют единый backend pipeline.

## 6. Требования к тестированию

### 6.1. Общий подход к тестированию

Для проекта должно быть предусмотрено тестирование основных сценариев работы системы. Проверка должна включать как ручные тест-кейсы, так и базовые автотесты для ключевых компонентов. Проверка должна покрывать все основные режимы работы системы: распознавание договора, анализ договора и сравнение двух договоров.

---

### 6.2. Ручное тестирование

Ручное тестирование должно покрывать основные пользовательские сценарии:

- загрузка корректного PDF-документа  
- попытка загрузки невалидного файла  
- успешное извлечение текста из PDF  
- корректная деперсонализация чувствительных данных  
- успешный анализ документа через LLM  
- формирование аналитического отчета  
- генерация итогового `.docx` файла  
- корректное отображение статусов в режиме анализа договора  
- корректное отображение статусов в режиме сравнения двух договоров  
- корректное уведомление пользователя о запуске OCR  
- корректное логирование выбранного режима обработки  
- корректное логирование этапов сравнения двух документов  
- вывод дисклеймера о справочном характере отчета  
- успешное сохранение результата в Google Drive  
- корректная обработка ошибок и уведомление пользователя
- успешная обработка сканированного PDF-документа с использованием OCR  
- корректное извлечение текста из сканированного документа 
- успешная обработка DOCX-документа в режиме анализа
- успешная обработка двух документов в режиме сравнения
- успешная обработка двух PDF-документов в режиме сравнения
- успешная обработка двух DOCX-документов в режиме сравнения
- успешная обработка смешанного сценария сравнения (PDF + DOCX)
- корректное отображение пользователю выбора режима работы
- корректное отображение запроса на возврат результатов распознавания при сравнении PDF-документов
- корректный выбор способа получения распознанных результатов:
  - отправка в чат
  - сохранение в Google Drive
- корректная работа OAuth-авторизации пользователя для подключения Google Drive
- корректное сохранение итоговых файлов в Google Drive пользователя после OAuth-авторизации
- корректная обработка сценария, когда пользователь не прошел OAuth-авторизацию, но выбрал сохранение в Google Drive
- корректное логирование использования OCR в консоль
- корректное формирование сравнительного отчета по двум договорам
- корректное формирование аналитического отчета в режиме анализа без генерации `.docx`

---

### 6.3. Автоматизированное тестирование

В рамках MVP должны быть реализованы автотесты для наиболее критичных и изолированных компонентов системы, в том числе:

- модуля деперсонализации данных  
- функций выделения и замены чувствительных данных  
- вспомогательных функций обработки текста  
- генерации структурированных выходных данных  
- отдельных этапов пайплайна, где это возможно без сложной внешней интеграции  
- базовая проверка корректности работы OCR-модуля (при наличии изолируемых функций)  
- обработка входных форматов для разных режимов работы (PDF / DOCX)
- базовые тесты маршрутизации по выбранному режиму работы
- базовая проверка логики сравнения двух документов
- базовая проверка формирования аналитического отчета
- базовая проверка логики выбора возврата распознанных результатов
- базовая проверка логики выбора способа доставки результата (чат / Google Drive)
- базовая проверка обработки статусов задачи, включая запуск OCR
- базовая проверка обработки ошибок интеграции с Google Drive

---

### 6.4. Ограничения тестирования MVP

Для MVP не требуется полное покрытие системы автоматизированными тестами. Основной акцент должен быть сделан на проверке критических сценариев, стабильности базового потока обработки и отсутствии критических ошибок при демонстрации проекта. В рамках MVP допускается ограниченное покрытие автотестами внешних интеграций (LLM API, Google Drive OAuth, Telegram Bot API), при этом основные пользовательские сценарии должны быть проверены вручную.

---

## 7. Сценарии использования

### 7.1. Основной сценарий (обработка договора)

1. Пользователь открывает Telegram-бота  
2. Пользователь отправляет PDF-документ  
3. Система подтверждает получение файла  
   3.1. Система предлагает пользователю выбрать, сохранять ли результат в Google Drive  
   3.2. Пользователь выбирает опцию сохранения  
4. Система уведомляет о начале обработки  

Далее пользователь получает статусы:

- извлечение текста (или OCR для сканированного документа)   
- деперсонализация данных  
- анализ документа  
- формирование отчета  
- генерация файла  

1. Система завершает обработку  
2. Пользователь получает:
   - краткий аналитический отчет  
   - файл в формате `.docx`  
   - (при включенной опции) ссылку на файл в Google Drive  
3. Система отображает дисклеймер о справочном характере отчета  

---

### 7.2. Сценарий с ошибкой (невалидный файл)

1. Пользователь отправляет файл  
2. Система определяет, что файл не является PDF  
3. Система уведомляет пользователя об ошибке  
4. Пользователю предлагается отправить корректный файл  

---

### 7.3. Сценарий ошибки обработки

1. Пользователь отправляет PDF-документ  
2. Система начинает обработку  
3. На одном из этапов возникает ошибка  
4. Система:
   - логирует ошибку  
   - уведомляет пользователя о проблеме  
5. Пользователь может повторить попытку  

---

### 7.4. Сценарий сохранения в Google Drive

1. Пользователь отправляет PDF-документ
2. Пользователь выбирает опцию сохранения результата в Google Drive    
3. Система выполняет обработку  
4. После генерации результата система сохраняет файл в Google Drive  
5. Пользователь получает ссылку на файл  

---

### 7.5. Сценарий отображения статусов

1. Пользователь выбирает режим работы системы  
2. Загружает один или два документа  
3. Система последовательно отправляет сообщения о ходе обработки  
4. Набор статусов зависит от выбранного режима:
   - распознавание договора  
   - анализ договора  
   - сравнение договоров  
5. Если для документа требуется OCR, система явно уведомляет пользователя о запуске распознавания  
6. По завершении система уведомляет пользователя о готовности результата или о возникновении ошибки  

---

### 7.6. Сценарий обработки сканированного документа (OCR)

1. Пользователь отправляет PDF-документ  
2. Система принимает файл  
3. Система определяет отсутствие текстового слоя  
4. Система применяет OCR для извлечения текста  
5. Далее выполняется стандартный сценарий обработки:
   - деперсонализация  
   - анализ через LLM  
   - генерация отчета  
   - формирование `.docx`  
6. Пользователь получает результат  

---

### 7.7 Сценарий сравнения двух договоров

1. Пользователь выбирает режим сравнения  
2. Загружает два документа  
3. Если документы в PDF, система предлагает:
   - вернуть ли результаты распознавания  

4. Если пользователь выбирает "да":
   4.1. Система запрашивает способ получения:
        - в чат  
        - в Google Drive  

5. Система выполняет:
   - извлечение текста (или OCR)  
   - деперсонализацию  
   - сравнительный анализ  

6. Пользователь получает:
   - отчет о различиях  
   - (опционально) распознанные документы  

---

## 8. Требования к интерфейсу

### 8.1. Формат интерфейса

Пользовательский интерфейс реализуется в виде:

- Telegram-бота  
- Web-интерфейса

Интерфейс должен обеспечивать:
- простой вход в сценарий обработки документа
- понятную последовательность действий
- информирование пользователя о текущем статусе обработки
- получение итоговых результатов в том же чате

---

### 8.2. Основные элементы взаимодействия

Интерфейс должен поддерживать следующие действия пользователя:

- запуск бота
- отправка PDF-документа
- выбор опции сохранения результата в Google Drive перед началом обработки
- возможность подключения Google Drive через OAuth (по запросу пользователя)  
- получение статусов выполнения
- получение итогового результата

---

### 8.3. Основные команды и сценарии

На уровне MVP интерфейс должен поддерживать:

- `/start` — запуск бота и краткое описание возможностей
- выбор режима работы:
  - распознавание договора  
  - анализ договора  
  - сравнение договоров  
- отправку PDF-файла в чат
- выбор действия через кнопки или простой интерфейс Telegram:
  - сохранить результат в Google Drive
  - кнопка "Подключить Google Drive" для прохождения авторизации  
  - не сохранять результат в Google Drive

При необходимости также может быть добавлена команда:
- `/help` — краткая инструкция по использованию

---

### 8.4. Поведение интерфейса при обработке файла

После получения PDF-документа интерфейс должен:

1. подтвердить получение файла
2. предложить выбрать, нужно ли сохранять результат в Google Drive
3. после выбора опции запустить обработку документа
4. последовательно сообщать пользователю о текущих этапах обработки

Пример этапов:
- файл получен
- извлечение текста (или OCR при необходимости)  
- деперсонализация данных
- анализ документа
- генерация отчета
- формирование `.docx`
- сохранение в Google Drive (если выбрана опция)
- обработка завершена

5. Набор отображаемых этапов зависит от выбранного пользователем режима работы системы.
6. Если для PDF-документа отсутствует текстовый слой, интерфейс должен явно сообщать пользователю о запуске OCR.

---

### 8.5. Формат сообщений пользователю

Сообщения должны быть:
- короткими
- понятными
- последовательными
- без избыточной технической информации

Интерфейс должен показывать:
- статус обработки
- сообщение об успешном завершении
- сообщение об ошибке, если обработка завершилась неуспешно
- дисклеймер о том, что аналитический отчет носит справочный характер и не является юридической консультацией

---

### 8.6. Формат выдачи результата

После завершения обработки пользователь должен получить:

- краткий аналитический отчет в текстовом виде
- итоговый файл в формате `.docx`
- ссылку на файл в Google Drive (если была выбрана соответствующая опция)

---

### 8.7. Требования к UX

Интерфейс должен:
- быть интуитивно понятным
- не требовать от пользователя сложной настройки
- не оставлять пользователя без обратной связи во время обработки
- явно показывать, что файл принят в работу
- явно показывать завершение обработки или возникновение ошибки

---

## 9. Ограничения и допущения

### 9.1. Функциональные ограничения

- система поддерживает PDF-документы и DOCX-документы (для режимов анализа и сравнения)  
- система ориентирована на обработку одного типа документов — договоров  
- система не выполняет полное восстановление форматирования исходного документа  
- система не поддерживает редактирование документов внутри интерфейса  

---

### 9.2. Ограничения MVP

- система обрабатывает один запрос за раз  
- не реализована очередь задач  
- не реализована многопользовательская конкурентная обработка  
- отсутствует хранение истории обработки документов  
- отсутствует база данных  

---

### 9.3. Ограничения точности

- извлечение данных может быть неточным в зависимости от качества PDF  
- результат работы LLM не гарантирует 100% точность  
- аналитический отчет может содержать упрощения или неточности  
- система не заменяет юридическую экспертизу
- OCR может работать неточно при низком качестве сканов или изображений    

---

### 9.4. Ограничения по безопасности данных

- система не сохраняет персональные данные пользователей  
- обработка выполняется в оперативной памяти  
- персональные данные заменяются на маркеры перед отправкой в LLM  
- исходные данные не передаются во внешние сервисы без деперсонализации  

---

### 9.5. Технические ограничения

- корректная работа зависит от стабильности внешних API (LLM, Google Drive)  
- скорость обработки зависит от размера документа и времени ответа LLM  
- качество извлечения текста зависит от структуры PDF  
- качество OCR зависит от качества исходного документа (скан, фото, освещение, разрешение)  

---

### 9.6. Допущения

- пользователь предоставляет корректные PDF-документы  
- документ может содержать текстовый слой или быть сканированным  
- для сканированных документов используется OCR-обработка   
- пользователь понимает, что система предоставляет вспомогательный результат  
- пользователь самостоятельно проверяет итоговый результат перед использованием  

## 10. Риски проекта

### 10.1. Риски качества данных

- низкое качество исходных PDF-документов может привести к ошибкам извлечения текста  
- OCR может работать неточно при плохом качестве сканов (размытые изображения, низкое разрешение, шум)  
- сложная структура документа (таблицы, нестандартная верстка) может быть обработана некорректно  

---

### 10.2. Риски, связанные с LLM

- возможны неточности или галлюцинации в ответах LLM  
- LLM может некорректно интерпретировать юридические формулировки  
- результаты анализа могут отличаться при повторных запросах  

---

### 10.3. Риски внешних интеграций

- возможны задержки или ошибки при работе с LLM API  
- возможны сбои при работе с Google Drive API  
- зависимость от доступности внешних сервисов  

---

### 10.4. Риски производительности

- увеличение времени обработки при больших документах  
- задержки из-за времени ответа LLM  
- увеличение времени обработки при использовании OCR  

---

### 10.5. Риски безопасности

- потенциальный риск утечки персональных данных при некорректной реализации деперсонализации  
- необходимость строгого контроля обработки чувствительных данных  

---

### 10.6. Риски MVP

- ограниченная функциональность может не покрывать все реальные кейсы  
- отсутствие обработки сложных документов (например, с большим количеством таблиц)  
- отсутствие хранения истории обработки  

---

### 10.7. Риски пользовательского опыта

- пользователь может не понимать ограничения системы  
- ожидание мгновенного результата при фактической задержке обработки  
- недоверие к результатам анализа без пояснений  

---

### 10.8. Меры снижения рисков

Для минимизации рисков в рамках MVP предусмотрены следующие меры:

- отображение статусов обработки пользователю  
- добавление дисклеймера о справочном характере отчета  
- деперсонализация данных перед отправкой в LLM  
- логирование всех этапов обработки  
- ограничение области применения (договоры)
